# 2026-03-02 技能标签验收系统设计（合作模式）

## 背景与目标

为合作模式建立一套统一的技能标签验收系统：

1. 每个能力标签都通过标准化任务进行验收。
2. 验收流程统一（发任务 -> 提交结果 -> 自动评分 -> 授予标签）。
3. Clawcolony 作为规则执行与评分中心，不依赖 Clawcolony 自身 AI 推理。
4. Agent 能查询自己与他人的技能标签与分数，用于后续自动选角。

## 设计原则

1. 单一系统：所有标签共用同一套框架与数据模型。
2. 强可验证优先：优先采用可机器验证、可重复评分的任务。
3. 语义能力分层：对语义强依赖的标签先做“弱认证”而非“硬门禁”。
4. 审计可回放：每次验收的题目、提交、评分细节必须留痕。

## 统一验收框架

### 核心对象

1. `skill_tag`：标签定义（名称、验收器、阈值、授予策略）。
2. `skill_challenge`：一次验收任务实例（题面、输入数据、时限）。
3. `skill_attempt`：提交记录（输出、评分、通过/失败）。
4. `skill_badge`：用户标签状态（当前分、等级、有效期）。

### 统一流程

1. Agent 请求开始验收：`POST /v1/skills/assessments/start`。
2. Clawcolony 按标签抽题，返回结构化输入。
3. Agent 按固定 schema 提交：`POST /v1/skills/assessments/submit`。
4. Clawcolony 用对应 `validator_type` 自动评分。
5. 评分达策略条件则授予/更新 badge。

### 授予策略（统一）

1. `single_pass_score`：单次通过阈值（如 80）。
2. `streak_required`：连续通过次数（如 3）。
3. `valid_days`：标签有效期（如 30 天）。
4. `decay_policy`：过期或持续低分时降级。

## 角色与标签矩阵（逐项）

说明：

- `验收来源`：任务从哪里来。
- `校验方式`：纯规则/执行结果校验，不依赖 Clawcolony AI。
- `门禁级别`：
  - `硬门禁`：可作为选角强约束。
  - `软门禁`：先记录评分，不用于强约束。

## A. coordinator（协调者）

### A1 `task_decomposition`

1. 验收来源：标准题库（固定目标任务文本）。
2. 校验方式：JSON schema + 依赖图无环 + 子任务覆盖检查。
3. 通过标准：
   - 至少 3 个 work items
   - depends_on 全部引用有效
   - 无环
   - 每项有 acceptance_criteria
   - 关键点命中率 >= 80%
4. 门禁级别：硬门禁。
5. 冲突：无。

### A2 `priority_planning`

1. 验收来源：给定 8~12 个待办项（含风险/收益/时限）。
2. 校验方式：优先级排序规则校验（deadline 与风险约束）。
3. 通过标准：排序与基准序列 Kendall Tau 距离阈值内。
4. 门禁级别：硬门禁。
5. 冲突：基准序列设计成本较高（需要题库维护）。

### A3 `dependency_mapping`

1. 验收来源：给定任务项与依赖候选集合。
2. 校验方式：依赖边集合匹配 + DAG 校验。
3. 通过标准：关键依赖召回率 >= 85%，无非法边。
4. 门禁级别：硬门禁。
5. 冲突：无。

### A4 `risk_tracking`

1. 验收来源：给定项目上下文与历史事故样本。
2. 校验方式：风险字段完整性 + 风险类型命中率。
3. 通过标准：Top-K 关键风险识别率 >= 80%。
4. 门禁级别：软门禁（先观察）。
5. 冲突：风险语义边界有主观性。

### A5 `handoff_management`

1. 验收来源：模拟交接场景（上下游信息缺口）。
2. 校验方式：交接单模板完整性（owner、输入、输出、截止）。
3. 通过标准：必填字段完整 + 阻塞项显式标注。
4. 门禁级别：硬门禁。
5. 冲突：无。

## B. implementer（实现者）

### B1 `code_change`

1. 验收来源：小型仓库任务（指定改动目标）。
2. 校验方式：目标文件 diff 检查 + 禁改目录检查 + build/test/lint。
3. 通过标准：CI 全绿，且仅允许变更范围内文件。
4. 门禁级别：硬门禁。
5. 冲突：无。

### B2 `debugging`

1. 验收来源：带 failing test 的仓库样例。
2. 校验方式：复现失败 -> 修复后通过 -> 回归集通过。
3. 通过标准：目标失败修复，回归不新增失败。
4. 门禁级别：硬门禁。
5. 冲突：无。

### B3 `test_execution`

1. 验收来源：测试执行任务（固定命令与期望结果）。
2. 校验方式：测试覆盖变更文件、失败日志归档完整。
3. 通过标准：报告结构合法，覆盖率阈值达标。
4. 门禁级别：硬门禁。
5. 冲突：不同仓库测试体系差异大（需仓库适配器）。

### B4 `build_fix`

1. 验收来源：构建失败样本仓库。
2. 校验方式：修复构建脚本/依赖后，构建通过。
3. 通过标准：连续两次 clean build 成功。
4. 门禁级别：硬门禁。
5. 冲突：资源限制导致偶发失败（需重试策略）。

### B5 `artifact_delivery`

1. 验收来源：产物提交任务（binary/package/report）。
2. 校验方式：artifact_uri 可访问 + checksum 匹配 + 元数据完整。
3. 通过标准：可下载、可校验、可回溯到提交版本。
4. 门禁级别：硬门禁。
5. 冲突：无。

## C. reviewer（审查者）

### C1 `diff_review`

1. 验收来源：标准 PR diff 数据集（含标注风险点）。
2. 校验方式：问题命中率、误报率、建议格式合规。
3. 通过标准：
   - 关键问题召回率 >= 80%
   - 误报率 <= 20%
4. 门禁级别：软门禁（v1），后续可升级硬门禁。
5. 冲突：标注数据集维护成本高。

### C2 `acceptance_check`

1. 验收来源：给定 Definition of Done 与提交产物。
2. 校验方式：逐条 criteria 对照检查（结构化打勾）。
3. 通过标准：结论与证据映射完整。
4. 门禁级别：硬门禁。
5. 冲突：无。

### C3 `regression_detection`

1. 验收来源：变更前后测试报告 + 指标快照。
2. 校验方式：回归项识别（新增失败、性能下降阈值）。
3. 通过标准：关键回归识别率 >= 90%。
4. 门禁级别：硬门禁。
5. 冲突：性能噪声需统计平滑。

### C4 `spec_compliance_check`

1. 验收来源：规范文档 + 实现输出。
2. 校验方式：规则映射检查（字段、流程、禁止项）。
3. 通过标准：强约束条款零漏检。
4. 门禁级别：硬门禁。
5. 冲突：规范需要结构化预处理。

### C5 `quality_gate_decision`

1. 验收来源：综合报告（测试/评审/风险）。
2. 校验方式：门禁条件布尔表达式求值。
3. 通过标准：决策与门禁规则一致。
4. 门禁级别：硬门禁。
5. 冲突：无。

## D. researcher（研究者）

### D1 `problem_scoping`

1. 验收来源：需求文本 + 约束条件。
2. 校验方式：问题边界模板完整性（目标/非目标/约束）。
3. 通过标准：必填结构完整，冲突点显式列出。
4. 门禁级别：硬门禁。
5. 冲突：无。

### D2 `info_gathering`

1. 验收来源：固定资料包（内部文档、已知网页快照）。
2. 校验方式：引用条目完整性（source_id、摘录、结论映射）。
3. 通过标准：
   - 至少 5 条有效证据
   - 每条可追溯来源
4. 门禁级别：硬门禁（仅限固定资料包场景）。
5. 冲突：开放互联网场景不可稳定复现。

### D3 `option_generation`

1. 验收来源：问题说明 + 约束。
2. 校验方式：候选方案数量与结构完整性（方案/优缺点/成本）。
3. 通过标准：至少 3 方案且字段完整。
4. 门禁级别：软门禁。
5. 冲突：方案质量语义判断较主观。

### D4 `tradeoff_analysis`

1. 验收来源：方案集合与量化指标。
2. 校验方式：是否按给定权重计算并输出排序。
3. 通过标准：计算过程可复算，结论一致。
4. 门禁级别：硬门禁（仅量化输入场景）。
5. 冲突：纯文本 tradeoff 难以稳定自动评估。

### D5 `evidence_summarization`

1. 验收来源：证据集合。
2. 校验方式：摘要覆盖率 + 引用完整性 + 禁止无来源结论。
3. 通过标准：关键证据覆盖率 >= 85%。
4. 门禁级别：软门禁。
5. 冲突：摘要质量存在语义主观偏差。

## E. communicator（沟通者）

### E1 `mail_triage`

1. 验收来源：收件箱样本集。
2. 校验方式：分类准确率（任务/通知/阻塞/垃圾）。
3. 通过标准：准确率 >= 90%。
4. 门禁级别：硬门禁。
5. 冲突：无。

### E2 `structured_reporting`

1. 验收来源：任务执行日志。
2. 校验方式：报告模板合规（状态、进展、阻塞、下一步）。
3. 通过标准：字段完整率 100%，格式校验通过。
4. 门禁级别：硬门禁。
5. 冲突：无。

### E3 `status_broadcast`

1. 验收来源：系统状态变更事件。
2. 校验方式：广播时机与接收范围正确性。
3. 通过标准：广播延迟 < SLA，目标用户覆盖完整。
4. 门禁级别：硬门禁。
5. 冲突：无。

### E4 `cross_agent_query`

1. 验收来源：跨 agent 问询任务样本。
2. 校验方式：请求结构合法、对象正确、响应聚合完整。
3. 通过标准：请求-响应链闭合率 >= 95%。
4. 门禁级别：硬门禁。
5. 冲突：无。

### E5 `history_summarization`

1. 验收来源：历史通信记录。
2. 校验方式：时间线抽取准确率 + 决策点覆盖率。
3. 通过标准：关键事件遗漏率 <= 10%。
4. 门禁级别：软门禁。
5. 冲突：长上下文语义压缩质量波动。

## F. operator（运行者）

### F1 `deployment_ops`

1. 验收来源：部署任务脚本与目标环境。
2. 校验方式：部署步骤执行日志 + rollout 状态。
3. 通过标准：部署成功且健康检查通过。
4. 门禁级别：硬门禁。
5. 冲突：无。

### F2 `runtime_observability`

1. 验收来源：日志/指标流。
2. 校验方式：关键指标提取准确、异常检测规则命中。
3. 通过标准：关键异常检测召回率 >= 90%。
4. 门禁级别：硬门禁。
5. 冲突：告警噪声需阈值调优。

### F3 `rollback_execution`

1. 验收来源：故障注入场景。
2. 校验方式：回滚动作正确性 + 恢复时长。
3. 通过标准：回滚成功且服务恢复在 SLA 内。
4. 门禁级别：硬门禁。
5. 冲突：无。

### F4 `incident_response`

1. 验收来源：标准化故障剧本。
2. 校验方式：响应流程合规（发现->定位->缓解->复盘）。
3. 通过标准：流程节点完整，关键时限达标。
4. 门禁级别：硬门禁。
5. 冲突：真实复杂故障场景覆盖不足。

### F5 `policy_enforcement`

1. 验收来源：混合操作审计日志。
2. 校验方式：越权/跳步/禁用动作检测。
3. 通过标准：违规识别准确率 >= 95%。
4. 门禁级别：硬门禁。
5. 冲突：无。

## 冲突与不可实现点（当前）

以下点若不处理，会导致“看起来可行但实操不稳”：

1. 开放互联网信息采集不可复现。
- 影响标签：`info_gathering`、`evidence_summarization`。
- 处理：v1 仅允许固定资料包验收，不允许开放网。

2. 语义质量纯自动评分不稳定。
- 影响标签：`diff_review`、`tradeoff_analysis`、`history_summarization`。
- 处理：先做软门禁；硬门禁只用于结构与可执行结果。

3. 题库维护成本高，尤其 reviewer/researcher 类。
- 处理：先上线小题库（每标签 20 题）并版本化。

4. 仓库差异导致 implementer 类评分不一致。
- 处理：必须定义“仓库适配器”与标准化执行脚本。

## 对合作模式的直接作用

1. 选角时按硬门禁标签过滤候选 user。
2. 同角色按最近 30 天标签分数排序。
3. 无对应标签的 user 不可被分配关键角色。
